# url-to-screenshot implementation review — verified findings

Three review systems reconciled: 7-dimension Claude panel (adversarial verify) + parent harness run. All findings below verified against the real code.

**Ship verdict: needs-fixes-before-ship**

The url-to-screenshot skill's offline-safety surface (SSRF admission gate, scheme allow-list, redact_url, metadata host/IP denylist, fail-closed resolution) is well-architected and the offline harness gates are GREEN. The selftest-exercised decision layer is solid and honest about Tier-1's documented-unmitigated redirect SSRF. However two HIGH findings block ship because they are NOT documented intentional limitations — they are internal contradictions between the plan-mandated BLOCKING Linux Chromium capture job and the shipped engine, plus agent-facing prose that promises a PNG the capture verb never produces. Both were reproduced end-to-end: file:// is rejected as BLOCKED_SCHEME (so the blocking CI job aborts on its first command under set -euo pipefail) and run_capture returns READY with no screenshot, deferring to a non-existent engine entry point. A separate medium SSRF gap lets the Alibaba metadata IP through in its IPv4-mapped IPv6 form under --allow-private-targets, breaking the explicitly-promised UNCONDITIONAL metadata guarantee. The remaining six findings are low-severity doc-precision and test-meaningfulness nits. Verdict: needs-fixes-before-ship — the two HIGH self-contradictions must be resolved (either wire the engine or honestly demote/reword), and the metadata-denylist invariant should be restored to match its own documentation.

## Blocking

### [high] Plan-mandated BLOCKING Linux Chromium capture job is dead-on-arrival against the shipped engine (file:// rejected + capture is a no-PNG stub)
- Evidence: Reproduced verbatim. (1) security.py:158-159 raises TargetBlocked(BLOCKED_SCHEME) for any non-http(s) scheme; isolated probe: validate_target_url('file:///tmp/plain.html') -> BLOCKED BLOCKED_SCHEME. cmd_capture (url_to_screenshot_runtime.py:55-61) catches TargetBlocked and returns exit 2. The follow-up `verify --png` (tests.yml:110) reads a missing file. This is an internal plan contradiction: §7 S1.1 mandates file: -> BLOCKED_SCHEME 'Not overridable', while §2/Phase 7/§5 T3-T5 require a BLOCKING job capturing file:// fixtures — the two cannot coexist.
- Fix: Pick one and make plan+code+CI consistent: (a) wire run_capture to actually launch (oneshot.run_oneshot for Tier-1 / a real CDP runner for Tier-2), allocate output via naming, write the PNG, AND add a narrow file:// admission allowance gated to the capture path for CI fixtures (leaving http/https SSRF rules unchanged); OR (b) demote the linux-capture job to non-blocking and state plainly in the plan/§7 that T3/T4-capture/T5-reap are NOT enforced on every PR (the plan's own §7/Phase-7 escape hatch at line 538). As written the blocking job cannot pass.

### [medium] IPv4-mapped IPv6 form of the Alibaba metadata IP bypasses the UNCONDITIONAL metadata denylist under --allow-private-targets
- Evidence: Reproduced against the live module. _is_metadata_ip (security.py:102-116) compares only literal/canonical METADATA_IPS via ipaddress equality and never unwraps ip.ipv4_mapped — asymmetric to _ip_is_blocked (security.py:95-98) which DOES unwrap. Because ipaddress.ip_address('::ffff:100.100.100.200') != ipaddress.ip_address('100.100.100.200') and the mapped form is_private=False/is_global=True, it relies SOLELY on the denylist. Isolated probe under allow_private=True: validate_target_url('http://[::ffff:100.100.100.200]/') -> ADMITTED ips=['::ffff:100.100.100.200']; the compressed alias 'http://[::ffff:6464:64c8]/' -> ADMITTED; plain v4 'http://100.100.100.200/' -> BLOCKED_METADATA_ENDPOINT. The leak is broader than first claimed: '::ffff:169.254.169.254' is ALSO admitted under allow_private (the link-local block at security.py:175-178 is exactly what `continue` skips). This breaks the explicitly-promised invariant at security.py:13-14/20/150-152/171 that the metadata denylist is 'unconditional, even under allow_private'. Same gap in revalidate_resolved_address (security.py:196-200), the redirect/sub-resource hook. selftest.py:123-128/140-142 only test 169.254.169.254 and metadata.google.internal, never 100.100.100.200 and never any IPv4-mapped form.
- Fix: In _is_metadata_ip, after the literal/canonical comparison, unwrap IPv4-mapped IPv6 and re-check: parse text to ip, and if getattr(ip,'ipv4_mapped',None) is not None, return _is_metadata_ip(str(ip.ipv4_mapped)). This restores the unconditional guarantee in both validate_target_url and revalidate_resolved_address. Add selftest cases for ::ffff:100.100.100.200 and ::ffff:169.254.169.254 with allow_private=True asserting BLOCKED_METADATA_ENDPOINT. Do not weaken any existing control.

## Strengths (verified)

- The SSRF admission gate is fail-closed and layered: unresolvable hosts are rejected (resolve_host raises BLOCKED_INPUT), the scheme allow-list rejects everything but http/https, and _ip_is_blocked correctly unwraps IPv4-mapped IPv6 and embedded forms for the private/loopback/link-local/multicast/reserved/0.0.0.0-8 checks — the metadata-IP path is the only spot that missed the unwrap.
- redact_url (security.py:203-218) correctly strips query, fragment, and user:pass@ userinfo while preserving scheme/host/port/path, so logs and result.json identify the page without leaking tokens, and it fails safe to '(unparseable-url)'.
- The offline harness posture is honest and disciplined: the selftest is explicitly kept out of the subprocess-bearing import graph, the dispatcher binds allow_private solely from the parsed flag (no env-var relaxation path exists anywhere — grep-confirmed), and the documented Tier-1 redirect/sub-resource SSRF limitation is flagged transparently rather than hidden.
- Independently confirmed all gate claims hold at the code level: the offline-safety flags, the no-env-relaxation wiring, and the metadata denylist for plain-v4/hostname forms all behave as documented — the defects are at the edges (IPv4-mapped form, capture-half wiring), not in the core admission logic.

### [low] installer/ai_agents_skills/runtime_smoke.py:677 (all-passed check)
- Problem: The all-passed check `payload.get("passed") == payload.get("total")` evaluates to True when BOTH the `passed` and `total` keys are absent, because `None == None` is True. This silently passes a payload missing both counters, contradicting the plan's explicit invariant that the branch use `payload.get(...) is True/False` semantics so a missing key fails rather than silently passing (tasks/u2s-plan-under-review.md:507-509). It is the one check in the branch that deviates from the documented missing-key-fails discipline used by the surrounding ok/status/failures/safety-flag checks.
- Evidence: runtime_smoke.py:677 `checks.append({"name": "all-passed", "ok": payload.get("passed") == payload.get("total")})`. Probe (run on a copy in scratch, not the canonical tree): a payload `{ok:true, status:ok, failures:[], smoke_mode:offline, network_required:false, live_api_attempted:false, package_install_attempted:false, server_started:false, browser_launched:false}` with `passed` and `total` BOTH OMITTED yields accepted=True, failed_checks=[]. Contrast: `{passed:5}` with total omitted correctly fails all-passed (None mismatch), and a missing `failures` key correctly fails no-failures (None == [] is False). Plan intent: tasks/u2s-plan-under-review.md:507-509 'using payload.get(...) is True/False (so a missing key fails rather than silently passing): ok is True, failures == [], passed == total'.
- Fix: Require both keys to be present integers before comparing, e.g. `passed = payload.get("passed"); total = payload.get("total"); ok = isinstance(passed, int) and isinstance(total, int) and passed == total`. This closes the None==None gap and matches the documented missing-key-fails contract. Optionally add a regression test asserting a payload omitting both `passed` and `total` (but otherwise ok:true/status:ok/failures:[]) FAILS the validator — none of the three existing rejection tests isolates this case (each corrupts a different key), and the M5 test only checks branch existence by string match, not key semantics.
